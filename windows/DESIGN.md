# NetDisplay 界面设计（统一版）

> 两版被否后重做。用户原话：「完全看不懂，怎么又出来一个自动」「更扯淡，我到底怎么启用中转？
> 如果我用中转，还要设置本机地址吗？」——**问题不在实现，在我们把实现细节当成了用户的选择题。**
> 本文定稿后两端统一照此实现。

## 一、两版失败的共同根因

| 版本 | 做法 | 为什么错 |
|---|---|---|
| v1 | `连接方式：自动 / 直连 / 中转` | **「自动」的存在本身就是证据**——我们自己也知道用户答不上「该选直连还是中转」，于是加第三个选项去掩盖，结果三个都要理解 |
| v2 | 去掉模式，改「监听 / 连接」，两个字段都填就并行竞速 | 用户看到「配对码」和「本机地址」两个框并列，**不知道该填哪个、为什么要填两个**；跨网络用中转时还在显示对方根本连不上的本机局域网地址 |

**根因：把 transport（直连/中转）当成了「用户要做的选择」。**
它其实是**程序自己能算出来的状态**——而且用户根本判断不了：我们自己都被 CGNAT 的相同出口 IP 骗过，以为两台机器同网。**既然我们都会判断错，就不该要求用户判断。**

## 二、设计原则

1. **用户只需理解一个概念：配对码。** 没有「直连/中转/自动」，没有 IP 输入框，没有模式选择。
2. **transport 是状态，不是选项。** 界面只显示「直连 3ms」或「中转 310ms」——这是信息，不是让人选的。
3. **参数就近、随时可改**，不必断开重连去翻设置。
4. **主界面永远一屏**，不滚动。高级项折叠。

参照 TeamViewer / AnyDesk / Parsec：它们都同时支持 P2P 与中转，**界面上从来没有这两个词**，用户只交换一个 ID。

## 三、界面

### 状态 1：未连接（启动即此屏）

```

│                                    │
│  我的配对码                         │
│  ┌──────────────────────────────┐  │
│  │   4 8 1 9 2 0        [换一个] │  │
│  └──────────────────────────────┘  │
│  把它告诉对方，对方输入后即可连上     │
│                                    │
│  ──────────── 或 ────────────      │
│                                    │
│  连接对方                           │
│  ┌──────────────────────────────┐  │
│  │  输入对方的配对码              │  │
│  └──────────────────────────────┘  │

**没有模式选择。** 本机**始终**在等对方输入我的码（后台常驻），同时你也可以输入对方的码去连。谁先动谁发起——这正是 AnyDesk 的模型，用户从不需要想「我该当哪一方」。

```

**局域网地址不再作为输入项存在**——它由程序在握手时自动交换（见下）。

## 四、底层：transport 怎么变成「状态」

```
1. 双方都连中转（一定能通）→ 撮合 → 建立会话
2. HELLO 里互相告知各自的局域网地址（新增可选字段 lanAddrs）
3. 后台并行尝试直连对方的 lanAddrs
4. 直连握手成功 → 无缝切换到直连，状态栏改显「直连 3ms」
5. 直连不通 → 继续用中转，状态栏显「中转 310ms」
```

- **升级只在待命时做**，投射中不切链路（避免画面中断）。
- 直连监听端口**始终开着**，不再依赖「监听模式」。
- 协议改动很小：HELLO 加可选 `lanAddrs`，靠 §0「容忍未知字段」规则天然向后兼容。

## 五、技术栈：换掉 Electron

用户要求不用 Electron。本机实测：**未装 .NET SDK**，但 **Rust 1.92 + cargo + MSVC 齐全**。

| 方案 | 体积 | 原生度 | 视频管线 | 评估 |
|---|---|---|---|---|
| Electron（现状） | 71 MB | 低 | WebCodecs 现成 | 用户已否 |
| **Tauri（Rust + WebView2）** | **~5 MB** | 中高 | **WebCodecs 可复用** | **推荐** |
| Rust + windows-rs 纯 Win32 | ~3 MB | 最高 | **要全部重写**（MF 编解码 + WGC 采集 + D3D 渲染） | 工作量最大 |
| C# WinUI/WPF | ~10 MB+ | 最高 | 同样要全部重写 | 还得先装 .NET SDK |

**推荐 Tauri**，理由：
- Rust 环境已就绪，立刻能开工，不必先装 SDK
- 体积从 71MB 降到约 5MB，启动快，不再捆绑整个 Chromium
- **WebView2 支持 WebCodecs**，意味着**已跨机验证过的视频管线（真 HEVC 4:2:2、1583 帧零错）可以原样保留**——这是目前最有价值、最不该推倒重来的资产
- UI 层仍是 HTML/CSS，但配合系统字体与 Windows 11 配色可以做到看不出区别

**需要你确认的一点**：如果你对 Electron 的意见是「HTML 做的界面本身就不像原生」，那 Tauri 只解决了体积和启动速度，没解决这一条——那就得走 windows-rs 纯原生路线，代价是视频管线全部重写（Media Foundation 编解码、Windows.Graphics.Capture 采集、D3D11 渲染），是几个工作日的量，且要重新做一遍跨机验证。

## 六、迁移顺序（确认后执行）

1. Tauri 骨架 + 上面三个状态的界面
2. 协议层直接搬（`protocol.js` 与会话逻辑是纯 JS，无 Electron 依赖）
3. 采集/编码：`desktopCapturer` → Tauri 侧用 `windows-capture` crate 或保留 getUserMedia
4. 托盘菜单：Tauri 原生托盘（比 Electron 的更贴近系统）
5. 打包验证 + 跨机回归（必须重跑真 4:2:2 那条链路）
