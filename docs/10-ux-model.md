# NetDisplay 交互模型（两端统一 UX 的唯一依据 / SoT）

> 像 `02-protocol.md` 是线上协议的 SoT 一样，本文是**两端 UI/交互**的 SoT。
> 两端必须按同一个模型做，只是控件形态不同（Mac 菜单栏 / Windows 面板）。

## 一、现状为什么乱（要改掉的）

- 两端都把「发送」和「接收」当成**两条独立流程**，每条又各带一套 `直连/中转` 选择 → 用户在多处重复设连接方式，而且「谁 listen 谁 dial / 谁 register 谁 join」这种实现细节**暴露给了用户**。
- Windows：顶部「连接模式」配的是**接收**，下面「发送端」又有**并列的**「▶ 直连发送 (:47800)」「☁ 中转发送」两个按钮 —— 发送的连接方式和接收的连接方式各设一套。
- Mac：菜单里 `开始投送` / `接收投射` / `模式(直连/中转)` / `中转设置` 散落各处；而且**接收只实现了中转，不能直连**。
- 结果：搞不清谁是目标、谁是来源、谁和谁配对、谁连谁。

### 更深一层的病根（2026-07-22 定论）：把 transport 当成了「用户要做的选择」
即便加了「自动」，用户还是骂——因为**问题不在选项排列，在于我们要求用户做一个他答不上的判断**：「我和对方在不在同一个网？」他不知道；我们（靠 TCP connect）也判断不了（CGNAT 相同出口 IP、Clash TUN 都会骗过我们）。「自动」这个词本身就是我们没想清楚的自白（承认这题没法让用户答、又不敢替他决定）。
**结论：`直连/中转` 不是用户选项，是程序算出来的状态。** 参考 TeamViewer / AnyDesk / Parsec：用户只交换一个码，直连还是中继由 app 自己探测，界面只把结果**显示**出来。下面的模型据此重写。

## 二、统一模型：**一次配对 → 常驻连接 → 随时切换谁投谁**

三个概念清晰分层（正好对应协议 v1.4「连接与投射解耦」，所以**协议不用改，这是纯 UI 层**）：

### 1) 配对（Pairing）——一次性，两端填一样
一对设备互相认识一次，之后持久免码。
- **中转配对**：中转服务器地址 + token（两端填同一个）+ 首次一个配对码。
- **直连配对**：对方地址（局域网 IP / USB4 网桥）。
- 状态：`未配对` / `已配对 ✓`（可解除配对）。

### 2) 连接方式（Transport）——配对层的**一个**设置，收发共用
`[自动] / 直连（同网/USB4，低延迟）/ 中转（跨网络）`。**只设一次，发送和接收都用它**，不再各设一遍。
- 自动：**并行**尝试直连与中转、**先成功者胜**（happy-eyeballs 风格，超时 ~1.5s）——避免直连不可达时让用户干等好几秒。
  - ⚠️ **胜出判据必须是「收到对端的协议应答」，不是「TCP connect 成功」**（Windows 实测的坑，两端都要遵守）：装了代理/VPN（Mihomo/Clash、尤其 **TUN 透明代理**）的机器上，`connect` 连一个**根本不可达**的地址也会成功——代理接管了出站 TCP。所以只有收到直连的 Sender `HELLO` 或中转的 `RELAY_PAIRED` 才算这条路真的通到 NetDisplay。竞速期用临时解析器，胜出后把已读出的帧补回正式解析器（别丢掉 Sender HELLO）。**Mac 端做 auto 时同样照此**（用户 Mac 就是 Clash TUN 模式）。

### 3) 角色开关（Role）——**一个开关，随时切**，用户不用想「谁连谁」
本机三态：
- **待命**：空闲，画面显示「等待投射…」；对方一旦开始投射，**自动切到「接收显示」**。
- **投射出去（来源）**：选投射源（整屏 / 某个窗口），把本机画面投给对方。
- **接收显示（目标）**：显示对方投来的画面（通常由"待命"自动进入，无需手动）。

一个「停止 / 弹回」按钮回到待命。

**规则**：
- 配对后**连接常驻**（见下「连接角色 / 选项 A」）；**在同一台机器上切换投射源（整屏↔窗口）、开关投射都不断连**（沿用 VIDEO_CONFIG）。
- **切换投射方向（换成另一台当来源）= 快速重建连接**（直连 <100ms、中转 ~1s，UI 提示「切换中…」）。
- 同一对里**同一时刻只有一个来源**。
- **连接编排由 app 自动处理**，用户看不到：中转=来源 `register`、目标 `join`；直连=来源 `listen(47800)`、目标 `dial`。

### 连接角色（选项 A —— 已定，**不改协议**）
Windows review 指出：当前协议里**连接角色与投射角色绑定**（`listen/register` 方=sender，`VIDEO_FRAME` 单向），所以字面「不断连切换投射方向」需要改协议（**选项 B**：连接后双方都可发帧 + `ROLE_SWITCH` 投射权协商，复杂度大、体验增量小）。**决策 = 选项 A**：接受「切换方向=重建连接」，但做到用户无感——用户在乎的是「不用想谁连谁」，不是「物理上不断连」。

选项 A 的机制（**两端已对齐；权威细则见 91「更新之三十四」，两端必须一致实现**）：
- **A 位 / B 位**：`listen/register` 方＝**A 位**（source-capable）；`dial/join` 方＝**B 位**。
- **默认角色 = deviceId 字典序（UTF-8 字节序，JS `<` 与 Swift `<` 对 ASCII UUID 一致）**：deviceId 较小的一方=A 位，较大的=B 位。**首次配对**还不知对方 deviceId → 临时规则：**谁点「投射本机」谁就是 A 位**；HELLO 交换后双方各自算出默认角色，**连同 pairSecret 一起持久化**，此后一律按默认角色。
- **待命（连接常驻）**：A 位持续 `listen/register`、B 位持续 `dial/join` 并自动重连；连接建立即完成 HELLO/HELLO_ACK，A 位立刻发 `PROJECTION_STATE{active:false}`。→「对方一投过来我自动显示」成立。
- **A 位投**＝直接投、无重连；**B 位投**＝断开→角色反转（B 改 A 位）→重建→投（<100ms / ~1s，提示「切换中…」）。
- **⚠️ 断线重连一律回默认角色**（防死锁的关键）：非主动切换的断线 → 双方丢弃反转状态 → 按 deviceId 默认角色重连。代价：断线前 B 在投则重连后回待命、用户需再点一次投射（**确定性优先于自动恢复**）。
- **抢投**：想投方先发 `CONTROL(0x21){action:"stop"}` 让当前来源转空闲，再（必要时反转重建）开投；**同时抢投** → 比 deviceId，**较小者胜**、较大者让出（与默认角色同源，两端裁决一致）。
- **直连**：双方配对时各记对端 IP；A 位 `listen(47800)`、B 位 `dial 对方IP:47800`，角色判定与中转相同。
- **UI 承诺**：「切换是一个开关的事」——**不**承诺物理不断连（诚实）。

### 画质设置的归属
分辨率 / 缩放 / 帧率 是**目标端（接收方）**请求的（HELLO.screen），**目标端设、来源端采纳**；码率来源端可覆盖。UI 上归到「作为目标时的画质」，**不要和"发送"混在一起**。

## 三、两端 UI 落地（同一套，控件形态不同）

**一个「连接设置」面板（两端字段一致）**：
`连接方式[自动/直连/中转]` →（直连填`对方地址`；中转填`服务器+token`）；`配对状态/配对码`；`作为目标时的画质[分辨率/缩放/帧率/码率]`。
一个 `[连接] / [断开]`。

**主控制（连接后）**：
`[● 投射本机 ▾（源：整屏 / 某窗口）]  [■ 停止/弹回]`。接收是自动的（对方投来即显示）。

**各端具体**：
- **Mac 菜单栏**：`连接设置…`（面板）→ `连接`/`断开`；连上后 `投射本机 ▸（源子菜单）`、`停止投射`；画质子菜单标注「（作为目标时）」。去掉散落的 `模式`/`中转设置`/独立 `接收投射` 入口——统一到「连接设置 + 角色开关」。**补：接收支持直连**（目标 dial 对方 :47800）。
- **Windows 面板**：一个设置面板（连接方式 + 直连/中转 + 画质）+ 一个 `连接`；主界面连上后就是 `投射本机 ▾ / 停止`。**去掉「▶ 直连发送 (:47800)」「☁ 中转发送」两个并列按钮**——合并成一个「投射本机」，连接方式由设置决定，不在按钮上区分。

## 四、迁移要点
- **直连接收两端都要支持**（目标 dial 对方 :47800）。Mac GUI 目前只做了中转接收，补直连。
- CLI 入口（`--host` / `--server` / `--secret`）**保持不变**，不受 UI 重构影响。
- 协议 **不改**（v1.4 已解耦连接与投射）。
