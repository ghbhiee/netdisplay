# J1 跨机验收清单（新界面）

两端都重做完界面后跑这一遍。**两端照同一份脚本走**，否则出了分歧会先吵「你是不是步骤不一样」。

前置：两端都是新版；relay 用 15.tokencv.com:47700；两端「中转设置」里填同一个 token；
两台机器在同一局域网（要验直连升级）。

## 记录方式

每一步都记**三样**：界面显示的文字、两端日志里的关键行、实际观感。
只记「通过」没有用——出问题时回看才知道当时到底是什么状态。

失败时**先别改代码**，先抓：两端完整日志、界面截图、`RECV_STATS`/`SEND_STATS`、
relay 上的房间状态。这个项目里最贵的几个 bug 都是「表面症状和真实原因隔得很远」，
现场没抓全就只能靠猜。

## 1. 首次配对

| # | 操作 | 期望 |
|---|---|---|
| 1.1 | A 端点「＋ 添加设备」→「随机生成」得到 6 位码 | 码显示成 `123 456` 分组样式 |
| 1.2 | B 端「＋ 添加设备」→ 输入**同一个码** → 配对 | 两端设备列表各出现一条，名字是对端的「本机名称」 |
| 1.3 | 看两端日志的 pairHash | **两端一致**，且等于 §3.7 算法算出的值 |

> 这一步失败最可能是 §3.7 推导没对齐。别急着看网络，先各自打印 pairHash 比一下。

## 2. 任意方向投射

| # | 操作 | 期望 |
|---|---|---|
| 2.1 | A 端选「整块屏幕」→「开始投射」 | A 显示「正在投射给 {B}」；**B 自动弹出接收窗口**，无需操作 |
| 2.2 | 看 B 的接收窗口标题与左下角徽标 | 标题「NetDisplay — {A} 的画面」；徽标「接收中 · 直连/中转 · Nms」 |
| 2.3 | 等 10s 看徽标 | 应从「中转」变成「直连」，延迟掉到个位数 ms |
| 2.4 | A 端点「停止」 | 两端回待命；**B 的接收窗口自动关闭**，不留空窗 |

> 2.3 不发生的话，先确认两端 HELLO 里的 `lanAddrs` **没有把虚拟网卡地址报出去**
> （Mihomo/Clash 的 TUN、VMware、WSL）。报了的话会「升级成功」但延迟不降——白升级。

## 3. 换方向

| # | 操作 | 期望 |
|---|---|---|
| 3.1 | 承接 2.1（A 投 B），此时 B 端点「开始投射」 | 短暂「切换中…」（约 0.9s），随后变成 B 投 A |
| 3.2 | 看 A 端 | A 的接收窗口弹出，原来的投射停止；A 不应同时既投又收 |
| 3.3 | 再换回去一次 | 同上，可反复，不应越换越慢或卡死 |

> 换向要拆连接重建。**断线后两端都必须回到默认角色**，否则可能双方都监听或都拨号。

## 4. 抢投

| # | 操作 | 期望 |
|---|---|---|
| 4.1 | A 投 B 期间，B 端也点「开始投射」 | 只有一个来源存活，另一端自动转为接收，不出现双向投射 |
| 4.2 | 两端**几乎同时**点「开始投射」 | 由角色编排裁决出唯一赢家，两端最终状态一致，不死锁 |

## 5. 单窗口投射

| # | 操作 | 期望 |
|---|---|---|
| 5.1 | A 端选一个程序窗口投射 | B 只看到那个窗口，不是整屏 |
| 5.2 | 把该窗口**最小化** | **不允许悄悄退回整屏**。要么保持，要么明确提示 |
| 5.3 | 拖动改变该窗口大小 | 走 VIDEO_CONFIG 改尺寸，画面不糊、不拉伸 |

## 6. 失败态文案

断掉网络 / 填错 token / 输错码，各看一次界面提示。
判据：**每条提示都要说清下一步做什么**，不能只说「失败」。

## 7. 持久性

| # | 操作 | 期望 |
|---|---|---|
| 7.1 | 两端都退出再重开 | 设备列表还在，**不用重新输码**，自动连上待命 |
| 7.2 | 「解除配对」后重开 | 设备消失，回到需要重新配对的状态 |
