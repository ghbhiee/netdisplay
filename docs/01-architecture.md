## 关键设计决策

### 1. 传输层：统一用 TCP

- opendisplay 验证了 TCP 在低延迟场景可行（USB/局域网下无丢包，TCP 无重传开销）。
- USB4 直连带宽 10–40 Gbps、RTT < 1ms，TCP 完全不是瓶颈。
- 中转模式走公网，TCP 简化了 relay 实现（纯字节管道）和 NAT 穿越问题（都是出站连接）。
- **不做 UDP/WebRTC**（v1 范围内）。若中转模式实测延迟不满意，v2 再评估 QUIC。
- 所有 TCP 连接设置 `TCP_NODELAY`（禁用 Nagle），这对延迟至关重要，两端都必须设置。

### 2. 连接方向与角色分离

**连接方向**（谁拨号）和**协议角色**（谁发视频）是分离的：

- 直连模式：Mac（Sender）监听 `:47800`，Windows（Receiver）拨号连入。
- 中转模式：两端都拨号连 relay，配对成功后 relay 变成透明字节管道。
- 无论哪种模式，TCP 建立后的应用层流程**完全一致**：双方交换 `HELLO`/`HELLO_ACK`，之后 Sender 推视频流。这样两种模式共享同一套协议代码，只有"建立连接"一步不同。

### 3. 视频：H.264 Annex-B，v1 不加密不封装容器

- 编码：VideoToolbox H.264，RealTime 模式，禁 B 帧，关键帧间隔 2 秒 + 响应 `REQUEST_KEYFRAME`。
- 每个关键帧前必须携带 SPS/PPS（Annex-B 格式内联），这样 Receiver 可以在任意关键帧开始解码，中途加入/重连不需要额外协商。
- 码率：直连模式默认 40 Mbps（可配 20–80）；中转模式默认 10 Mbps（受公网上行限制，可配 4–20）。
- HEVC/AV1、自适应码率是 M4 内容。

### 4. 分辨率协商：Receiver 决定虚拟屏尺寸

Windows 端在 `HELLO` 中上报自己的屏幕物理分辨率和期望帧率，Mac 端据此创建同尺寸虚拟屏（HiDPI 时上报逻辑分辨率 + scale）。这样画面 1:1 映射，无缩放模糊。分辨率协商细节见 `02-protocol.md` 的 HELLO 消息。

### 5. 中转模式：配对制，不做端口穿透

用户明确要求"两边配对，而不是设置端口穿透"。设计：

- Relay 是一个无状态小程序（Go），只做两件事：**配对撮合**和**字节转发**。它不理解视频协议。
- Sender 连上 relay 后注册，生成 **6 位数字配对码**并显示在 Mac 端界面/日志上。
- Receiver 连上 relay，输入同一配对码，relay 撮合成功后向双方发 `RELAY_PAIRED`，之后 relay 对这两条连接做双向 `io.Copy`，退化为透明管道。
- 配对码 5 分钟有效、一次性。配对成功后进入正常 `HELLO` 流程。
- **持久配对（M4）**：首次配对成功后，Sender 在 `HELLO_ACK` 中下发 32 字节 `pairSecret`，双方持久保存。此后双方注册/加入时改用 `SHA-256(pairSecret)` 作为房间标识，relay 自动撮合，无需再输码。

### 6. 安全边界（v1 的明确取舍）

- 直连模式：USB4 点对点网络 / 家庭局域网，v1 不加密。
- 中转模式：v1 流量对 relay 可见（relay 是用户自己的服务器，可接受）；链路上是明文 TCP。**键盘输入回传（M4）实装前，必须先给中转模式加上加密**：方案定为配对后用 pairSecret 派生密钥，对每个协议帧做 ChaCha20-Poly1305（Swift 用 CryptoKit，Node 用内置 crypto），nonce 用 8 字节方向前缀 + 4 字节递增计数。v1 视频-only 阶段可暂缓。
- Relay 端口 47700 暴露公网，程序必须限制：未配对连接 30 秒超时踢掉、配对码爆破限速（同 IP 每分钟最多 10 次 JOIN）、单房间只允许一对连接。

### 7. 服务发现

- v1：不做自动发现。直连模式由 Windows 端手动输入 Mac 的 IP（USB4 直连时固定为 `10.77.0.1`，可做成默认值一键连接）。
- v2（可选）：Mac 端用 NWListener 的 Bonjour 广播 `_netdisplay._tcp`，Windows 端 mDNS 发现。优先级低，因为 USB4 场景 IP 是固定的。

## 延迟预算（直连 60fps 目标）

| 环节 | 目标 |
|---|---|
| 捕获（ScreenCaptureKit 回调） | ~8ms（1 帧间隔内） |
| 编码（VideoToolbox 实时模式） | 3–8ms |
| 传输（USB4 / 千兆局域网） | < 1ms / 1–3ms |
| 解码（WebCodecs 硬解） | 3–8ms |
| 渲染（canvas + rAF） | ~8ms |
| **端到端合计** | **25–35ms（约 2 帧）** |

中转模式额外增加两段公网 RTT/2 + relay 转发（<1ms），同城约 +15–30ms。

## 失败与重连策略

- Receiver 断线：Sender 保留虚拟屏 60 秒（避免 macOS 窗口大搬家），期间等待重连；超时后销毁虚拟屏。
- Sender 到 relay 的注册连接断开：指数退避自动重连（1s 起，上限 30s），重新注册同一房间。
- 视频流中断恢复后，Receiver 主动发 `REQUEST_KEYFRAME`。
- 心跳：任一方向 10 秒未收到任何帧（含 PING/PONG）视为断线。PING 间隔 3 秒，由 Receiver 发起，Sender 回 PONG，Receiver 借此测量并显示 RTT。
